# gRPC Transport Implementation Plan

This plan outlines the steps to add a gRPC transport to FastMCP, allowing clients to interact with the server using Protobuf instead of JSON-RPC.

## 1. Protobuf Definition (`mcp_transport.proto`)

```proto
syntax = "proto3";

package fastmcp.v1;

import "google/protobuf/struct.proto";

service MCPServer {
  // Discovery
  rpc ListTools (ListRequest) returns (ListToolsResponse);
  rpc ListResources (ListRequest) returns (ListResourcesResponse);
  rpc ListPrompts (ListRequest) returns (ListPromptsResponse);

  // Execution - CallTool is a server-side stream to support progress reporting
  rpc CallTool (CallToolRequest) returns (stream CallToolResponse);
  rpc ReadResource (ReadResourceRequest) returns (ReadResourceResponse);
  rpc GetPrompt (GetPromptRequest) returns (GetPromptResponse);

  // Notifications (Server-to-Client)
  rpc WatchChanges (WatchRequest) returns (stream Notification);
}

message ListRequest {
  string cursor = 1;
}

message CallToolRequest {
  string name = 1;
  google.protobuf.Struct arguments = 2;
}

message CallToolResponse {
  oneof frame {
    ProgressUpdate progress = 1;
    ToolResult result = 2;
  }
}

message ProgressUpdate {
  double progress = 1;
  double total = 2;
}

message ToolResult {
  repeated Content content = 1;
  bool is_error = 2;
}

message Content {
  string type = 1;
  string text = 2;
}

// ... Additional messages for Resources and Prompts
```

## 2. Progress Reporting Mechanism

To support `ctx.report_progress(current, total)` without changing user tool code, we implement a bridge:

1. **Streaming RPC**: `CallTool` is defined as `returns (stream CallToolResponse)`.
2. **Session Adapter**: A `GRPCSessionAdapter` is created for each RPC call. It implements `send_notification(method, params)`.
3. **Mapping**: When a tool calls `ctx.report_progress`, FastMCP routes this to the adapter. The adapter converts the `notifications/progress` JSON-RPC params into a `ProgressUpdate` Protobuf message and yields it to the gRPC stream.
4. **Final Result**: Once the tool function returns, the final `ToolResult` is yielded as the last frame of the stream.

## 3. Implementation Roadmap

### Phase 1: The Session Bridge
Create a `GRPCSessionAdapter` that implements the minimum interface required by `FastMCP`'s `Context` and `Middleware`. 
- **Progress**: Map `notifications/progress` to gRPC stream frames.
- **Logging**: Map `notifications/message` (from `ctx.info()`, etc.) to gRPC metadata or a dedicated log field in the stream.

### Phase 2: The gRPC Servicer
Implement the `MCPServerServicer`. Each RPC method will:
1. Extract arguments from Protobuf.
2. Construct a `MiddlewareContext`.
3. Wrap the call in an `async with Context(fastmcp=self.fastmcp, session=adapter):` block.
4. Invoke the high-level `FastMCP` methods.
5. Map results (e.g., `ToolResult`) back into Protobuf messages.

### Phase 3: Transport Integration
Update `fastmcp.server.mixins.TransportMixin`:
- Add `run_grpc_async(host, port)` method.
- Update `run_async` to accept `transport="grpc"`.
- Implement a server runner using `grpcio` or `grpclib`.

### Phase 4: Middleware Compatibility
Review existing middleware (Logging, Auth, Rate Limiting). Ensure they can handle the `MiddlewareContext` generated by the gRPC path, which may lack some JSON-RPC specific fields.

### Phase 5: Notification Mapping
Implement the `WatchChanges` server-side stream. Hook into FastMCP's internal event system (via `AggregateProvider` or a new event bus) to push notifications to active gRPC stream observers.
